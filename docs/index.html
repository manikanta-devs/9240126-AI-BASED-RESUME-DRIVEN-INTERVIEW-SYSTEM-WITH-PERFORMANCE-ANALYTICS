Review high-yield questions examiners typically ask during final project presentations. Click on any question to view structured responses and technical justifications.

Q1: Project Goal What is the core contribution of this system?
▼

The system bridges the gap between static resume screening and behavioral interview prep. It parses a candidate's uploaded resume (PDF/DOCX/TXT) using NLP (spaCy) to extract skills, education, and experiences. It then uses Generative AI (Gemini 1.5 Flash) to generate structured, role-specific, and resume-aware technical/soft question sets, culminating in voice/video practice with detailed scores (accuracy, pacing, depth) and analytics trends.

Topics: System Value, Target Problem
Q2: Natural Language Processing Why did you choose spaCy over basic regex or NLTK?
▼

spaCy provides a production-grade NLP pipeline with pre-trained statistical models. We utilize the English pipeline (en_core_web_sm) for:

  • Named Entity Recognition (NER): Extracting companies (ORG), dates (DATE), and education degrees.
  • Part-of-Speech (POS) Tagging: Understanding sentence structure for filtering nouns and verbs.
  • Phrase Matching: Running matcher logic against a dictionary of 800+ standard tech skills to ensure high extraction precision.

Unlike NLTK (which is academic/research focused) or Regex (which is fragile), spaCy parses resumes into clean structured JSON datasets in milliseconds.

Topics: NLP, Entity Extraction
Q3: Fallback Logic How does the system function if the LLM API key is missing?
▼

We designed the app to be highly resilient with a **Fallback Mode**:

  1. Question Generation: If Gemini fails/is unauthorized, the backend pulls from a local dataset of 100+ vetted questions filtered by candidate role.
  2. Answer Evaluation: Falls back to a deterministic heuristic script (backend/ai/answer_evaluator.py) that evaluates structural depth, keyword density, word counts, presence of structural patterns (like STAR method), and numeric details.

This guarantees the app remains completely operational during live evaluator demos even without active internet or credentials.

Topics: Resilience, Heuristics
Q4: UI/State Architecture Explain how you manage state and routing in the client app.
▼

We built a multi-layered state architecture:

  • React Context API (AppContext): Handles global state like API base configurations, health checks, and active session objects.
  • Zustand Store (store.js): Provides lightweight, reactive states for audio recordings, feedback logs, and visual settings.
  • React Router v6: Manages URL routes with ProtectedRoute layers ensuring user profiles are instantiated before interviews.

Routing uses lazy loading (lazy / Suspense) to split bundles, keeping the initial load under 300KB.

Topics: React, State, Lazy Loading
Q5: Security What security protections are implemented in the API?
▼

The backend includes multiple layers of defensive security:

  • Rate Limiting: Restricts endpoints using a memory-based bucket to prevent API scraping and LLM cost spikes.
  • Input Validation: Uses Pydantic validators (validators.py) to enforce schema checkups on inputs.
  • Security Headers: Integrates Werkzeug middleware to apply CSP (Content Security Policy), HSTS, and X-XSS protection.
  • File Upload Filters: Restricts uploads strictly to PDFs, DOCX, and TXT files, capped at 16MB.
Topics: Flask Security, Pydantic
